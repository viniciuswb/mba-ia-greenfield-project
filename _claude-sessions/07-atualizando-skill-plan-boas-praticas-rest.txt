      381 +### Response Body Rules                                       
      382 +                                                              
      383 +The body of a success response contains the **representation o
          +f the resource/data produced** or **nothing** (204). Never ret
          +urn `{ message: "..." }` as the sole content of a success resp
          +onse вҖ” the HTTP status code already communicates the outcome. 
      384 +                                                              
      385 +Decision guide:                                               
      386 +                                                              
      387 +- **Endpoint creates a resource** (register user, create video
          +) вҶ’ 201 + resource representation                             
      388 +- **Endpoint returns data** (login вҶ’ tokens, search вҶ’ results)
          + вҶ’ 200 + data                                                 
      389 +- **Endpoint executes an action without producing data** (logo
          +ut, confirm email, revoke token, send email) вҶ’ 204 with no bod
          +y

          +rmation вҖ” always returns the same response regardless of input
          + to avoid leaking information) вҶ’ 204 with no body (the absence
          + of a body naturally prevents information leakage)

      393 +                                                              
      394 +When the created resource has a retrieval endpoint (GET), incl
          +ude a `Location` response header pointing to the resource URI 
          +(e.g., `Location: /users/:id`). Omit when no GET endpoint exis
          +ts for the resource in the current phase.                     
      395 +

      397 +

          +s (login, logout, confirm, reset) are valid in REST. The rule:
      399 +

          +h the result                                                  
      401 +- Action that **triggers a side effect without producing data*
          +* (logout, confirm) вҶ’ 204 with no body
